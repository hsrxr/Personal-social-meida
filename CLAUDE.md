## 行为原则
### 查档求证，不臆猜接口
调用外部模块或 API 前，先查阅其文档或源码确认接口签名和行为。跨模块调用以 `docs/ARCHITECTURE.md` 第 3 节定义的接口契约为准，不凭记忆假设。

### 对齐需求再开工
实现前先确认需求理解一致。需求模糊时主动提问澄清，不自行脑补未明确的功能。

### 请示规则，不脑补业务
遇到业务规则不清楚的地方，先查文档或问人。不假设业务逻辑的工作方式。

### 复用存量，不新增冗余
优先复用现有模块和工具。新增代码前先检查是否有现成实现。

### 恪守架构规范
遵循 `docs/ARCHITECTURE.md` 定义的模块分层和数据流。架构调整前先讨论，不单方面偏离约定。

### 坦诚存疑，不懂就问
对不确定的事情和模糊的需求坦诚说明，不假装懂。涉及不熟悉的领域先查询验证再下结论。

### 分步迭代，不批量乱改
每次改动聚焦一个独立变更。不把多个不相关的修改混在同一个提交中。重构和功能开发分开进行。

## 代码原则
### 命名即文档
标识符反映意图而非实现细节。不为解释"怎么做的"写注释（代码本身应清晰），只为解释"为什么这么做"写注释。

### 可测性优先
函数只做一件事，副作用越少越好。依赖通过参数显式注入而非内部构造。纯函数优先，IO 操作推到边界层。

### 向内聚、向外松
模块公开能力（函数/接口）而非内部状态。跨模块通信通过明确的适配器或接口层，不共享内部数据结构。模块之间的依赖方向依赖稳定的抽象层。

### 无静默降级
禁止用 `??` 或 `||` 隐藏不应缺失的数据。违反数据契约时快速失败（throw）。降级仅在以下场景允许：面向用户的显示默认值（有明确设计意图）、有文档的可选配置、外部输入解析中缺失是合法状态的情况。

### 业务逻辑中不捕获所有异常
业务逻辑函数必须让错误自然向上传播。try/catch 仅限在 API/控制器边界、顶层事件处理器、或已定义部分失败恢复策略的场景。禁止为了返回 `null`/`undefined`/`false` 而捕获 `Error`。

### 测试必须在代码被删时失败
每条测试必须用具体值验证具体的业务结果。禁止作为唯一断言：`toBeDefined()`、`toBeTruthy()`、`toBeFalsy()`、`not.toBeNull()`。删除实现后测试仍然通过 = 无用测试。

### 不硬编码查询表
实现真实业务逻辑，而非硬编码匹配测试用例的返回值。使用多样化的测试数据和未出现在原始规格中的值来检测假实现。

### Bug 修复的红-绿顺序
先写能复现 bug 的失败测试（确认红），再修复代码（确认绿）。不可跳过红步骤。

### 调试日志和修复代码不混合同步
调试/诊断日志仅在人工确认修复生效后才移除。禁止在同一变更中同时修改调试日志和修复代码。
